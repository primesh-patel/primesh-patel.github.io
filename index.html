Key SQL Query (Window Function for KPI Trending)

WITH dept_kpi AS (
  SELECT dept_id, dept_name,
    DATE_FORMAT(report_week, '%Y-%u') AS week_label,
    SUM(kpi_value) AS weekly_total,
    AVG(kpi_value) OVER (
      PARTITION BY dept_id 
      ORDER BY report_week 
      ROWS BETWEEN 3 PRECEDING AND CURRENT ROW
    ) AS rolling_4wk_avg
  FROM operational_metrics
  WHERE report_week >= DATE_SUB(CURDATE(), INTERVAL 12 WEEK)
  GROUP BY dept_id, dept_name, week_label, report_week
)
SELECT *, 
  ROUND((weekly_total - rolling_4wk_avg) / rolling_4wk_avg * 100, 1) AS pct_vs_avg
FROM dept_kpi ORDER BY dept_id, week_label;

A complete analytics platform for Gurgaon's residential real estate market. Built to demonstrate end-to-end data science delivery — the kind of structured, phased project execution applicable to operational analytics in any domain.

The Problem

Real estate pricing in Gurgaon is opaque. Prices vary by sector, BHK, floor, furnishing, and dozens of other factors. The goal was a tool that predicted prices accurately and helped buyers and analysts understand market dynamics through data.

Data Collection & SQL Pipeline

Built a custom scraper using BeautifulSoup and Requests to collect 3,961 listings (2,880 flats + 1,081 houses from 99acres.com). Data loaded into MySQL for structured querying and validation.

-- Validate scraped data quality

AXA Insurance: Claim Prediction & Risk Scoring

December 2024· XGBoost Best Model· SHAP Explainability

🏆 ML classification pipeline to predict insurance claim likelihood for AXA — using SMOTE for class imbalance, XGBoost for modelling, and SHAP values for full underwriter-facing interpretability.

The Business Problem

Flag high-risk customers before claims occur, enabling proactive risk management. Core challenge: severe class imbalance with only ~8% of policies resulting in claims.

Data Analysis & EDA

Performed structured EDA to understand claim distribution, identify key risk factors, and visualize correlations. Used SQL to extract and validate the dataset before modelling.

-- Claim rate analysis by risk segment
SELECT 
  risk_category,
  vehicle_age_group,
  COUNT(*) AS policies,
  SUM(claim_flag) AS claims,
  ROUND(SUM(claim_flag)/COUNT(*)*100, 2) AS claim_rate_pct,
  AVG(annual_premium) AS avg_premium
FROM axa_policies
GROUP BY risk_category, vehicle_age_group
ORDER BY claim_rate_pct DESC;

Handling Class Imbalance with SMOTE

Used SMOTE (Synthetic Minority Oversampling Technique) combined with undersampling — dramatically improving recall without sacrificing precision. Evaluated multiple sampling ratios.

Model Development

XGBoost consistently outperformed Logistic Regression and Random Forest on AUC-ROC and F1. Hyperparameters tuned via Optuna for 100+ trials.

SHAP Explainability

SHAP values revealed driving history, vehicle age, mileage, and geographic zone as top predictors. Force plots enabled individual policy explanations — critical for trust in regulated insurance decisions.

Key Takeaways

  • Imbalanced classification requires deliberate strategy — accuracy alone is misleading
  • Always evaluate precision, recall, and F1 together for imbalanced problems
  • SHAP is essential for stakeholder trust in regulated industries
  • Hyperparameter optimization (Optuna) provides significant gains over default settings

Blog · Data Analytics Process

The 6-Step Data Analysis Process

January 2025· 10 min read· Google Data Analytics Framework

The Google Data Analytics Professional Certificate defines a structured 6-step process for data analysis: Ask → Prepare → Process → Analyze → Share → Act. Here's how I apply each step in real operational analytics work — with concrete examples from Power BI, SQL, and Python delivery.

Step 01
Ask — Define the Problem
Before touching any data, define exactly what question you're answering. In my KPI dashboard work, this meant structured workshops with stakeholders: "What does success look like for your team this quarter?" and "Which metrics do you check every Monday morning?"

Output: Documented KPI definitions, calculation logic, data owners, and reporting cadence. Without this step, you build the wrong dashboard perfectly.
Step 02
Prepare — Gather & Verify Data
Identify data sources, verify access, assess quality, and understand schema. In operational analytics, this means querying source databases to understand data freshness, completeness, and business rules.

SQL Example:
-- Data audit: assess completeness across source tables
SELECT 
  table_name,
  COUNT(*) AS total_rows,
  MIN(created_date) AS earliest_record,
  MAX(created_date) AS latest_record,
  SUM(CASE WHEN key_field IS NULL THEN 1 ELSE 0 END) AS null_key_count
FROM information_schema.tables t
JOIN source_tables s ON t.table_name = s.tbl
WHERE table_schema = 'operations_db'
GROUP BY table_name;
Step 03
Process — Clean & Transform
Data cleaning is 60–70% of analysis work. In my projects, I standardize formats, handle nulls, remove duplicates, reconcile values across sources, and apply business rules to calculated fields.
import pandas as pd
import numpy as np

# Operational data cleaning pipeline
def clean_ops_data(df):
    # Standardize date formats
    df['report_date'] = pd.to_datetime(df['report_date'], errors='coerce')
    
    # Remove duplicates (keep latest record per ID)
    df = df.sort_values('updated_at', ascending=False)
    df = df.drop_duplicates(subset=['record_id'], keep='first')
    
    # Fill nulls with business defaults
    df['dept_code'] = df['dept_code'].fillna('UNASSIGNED')
    df['metric_value'] = df['metric_value'].fillna(0)
    
    # Flag outliers (IQR method) without removing
    Q1 = df['metric_value'].quantile(0.25)
    Q3 = df['metric_value'].quantile(0.75)
    IQR = Q3 - Q1
    df['is_outlier'] = (df['metric_value'] < Q1 - 1.5*IQR) | (df['metric_value'] > Q3 + 1.5*IQR)
    
    return df
Step 04
Analyze — Model & Extract Insights
This is where patterns emerge. In operational analytics, analysis means KPI trending, variance analysis, cohort comparisons, and anomaly detection — not necessarily ML models. The question is always: "What does the data tell us that we didn't already know?"
Step 05
Share — Visualize & Report
Insights only create value when communicated clearly. In Power BI: choose the right visual for the message (trend = line chart, composition = bar, comparison = scatter), use consistent color coding (green = on track, amber = at risk, red = behind), and design for the audience's questions, not your analysis process.
Step 06
Act — Recommend & Deploy
Data analysis exists to drive decisions. Every report should end with clear, prioritized recommendations. In my reporting work, I always provide: "The data shows X. This means we should consider Y. The next step is Z by [owner] by [date]." Analytics without action is just expensive reporting.

Applying This to Real Work

Every KPI dashboard I've built follows this process. The stakeholders who get the most value are those who engage deeply in Steps 1 (Ask) and 6 (Act) — because data analysis is a loop, not a line. You act on insights, that creates new questions, and the cycle repeats.

Tools at Each Step

  • Ask: JIRA (requirements tracking), stakeholder interviews, KPI definition documents
  • Prepare: MySQL (data audits), GitHub (version control), Excel (data dictionaries)
  • Process: Python/Pandas (cleaning pipelines), SQL (ETL queries), Power BI (data model)
  • Analyze: SQL (aggregations, window functions), Python (statistical analysis), DAX (Power BI measures)
  • Share: Power BI (dashboards), Tableau (visualization), Python (automated reports via email)
  • Act: ServiceNow (tracking action items), JIRA (backlog updates), executive presentations
Blog · AI Tools

The AI Tools Every Data Analyst Should Know in 2025

March 2025· 8 min read

The analytics tooling landscape has changed dramatically. Knowing Power BI and SQL alone isn't enough — the real productivity multiplier is knowing which AI tools to layer on top of your core analytics workflow.

1. GitHub Copilot & Cursor

These handle boilerplate-heavy tasks effortlessly — SQL query templates, Python data cleaning pipelines, DAX measure drafts. Treat them like a fast junior analyst who needs clear instructions. I use Copilot for repetitive SQL patterns and Cursor for refactoring Python automation scripts.

2. Claude & ChatGPT for Analysis

LLMs are now integral for explaining statistical concepts, reviewing DAX logic, and drafting requirements documents. Claude handles long dashboards context without losing track — ideal for documentation and explaining analytical decisions to non-technical stakeholders.

3. AutoML: AutoGluon & FLAML

Tools like AutoGluon produce strong baseline models in minutes. Use them to benchmark what's achievable, then beat that baseline with domain knowledge and feature engineering.

4. Power BI Copilot

Microsoft's Copilot integration in Power BI now generates DAX measures from natural language descriptions — dramatically speeding up dashboard development. Still requires QA review, but saves significant time on first-draft measures.

5. Python Libraries: pandas-profiling & Great Expectations

Automated data quality profiling. pandas-profiling generates comprehensive EDA reports in one line. Great Expectations adds data validation rules to your pipelines — so you know when source data drifts from expected formats.

Conclusion

The best analysts aren't those who avoid AI tools — they use them without sacrificing rigour. Start with one, master it, then add the next. The fundamentals (SQL, data modelling, business context) still determine the quality of the output.

Project Report · AXA Insurance H1 Analysis

AXA Insurance H1 2007: Risk Analysis & Claims Prediction

December 2024· 10 min read

A machine learning system to predict insurance claim likelihood for AXA, identify high-risk customer segments, and produce interpretable outputs for underwriters — based on H1 2007 earnings and risk data analysis.

The Business Problem

Flag high-risk customers before claims occur. Core challenge: severe class imbalance with only ~8% of policies resulting in claims. Standard accuracy metrics are misleading in such scenarios.

AXA H1 2007 Context

Analysis grounded in AXA Group's H1 2007 performance data — examining revenue streams (Life & Savings, Property & Casualty, Asset Management) and claims patterns across geographies. Total revenues of €49.8bn with strong growth in emerging markets provided context for risk segmentation strategies.

Handling Imbalance with SMOTE

Used SMOTE combined with undersampling — dramatically improving recall without sacrificing precision. Evaluated multiple sampling strategies before settling on the optimal combination.

Model Development

XGBoost consistently outperformed Logistic Regression and Random Forest on AUC-ROC and F1. Tuned via Optuna for 100+ hyperparameter trials.

Explainability with SHAP

SHAP values revealed driving history, vehicle age, mileage, and geographic zone as top predictors. Force plots enabled individual policy explanations for underwriters — critical for regulated decision-making.

Key Takeaways

  • Imbalanced classification requires deliberate strategy — accuracy alone misleads
  • Always evaluate precision, recall, and F1 together for imbalanced problems
  • SHAP is essential for trust in regulated industries like insurance and banking
  • Business context (H1 earnings analysis) strengthens analytical framing and stakeholder communication
